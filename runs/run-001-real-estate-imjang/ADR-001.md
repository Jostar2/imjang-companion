# ADR-001

## Title

Bound property, visit, checklist, and report responsibilities for the v1 field-visit workflow

## Status

Accepted for `run-001-real-estate-imjang` on `2026-03-23`

## Context

- The PRD requires project/property CRUD, structured visit capture, browser-local autosave, comparison, and a shareable report view.
- The live codebase already exposes `projects`, `properties`, `visits`, `reports`, `auth`, `ops`, and attachment storage modules.
- The earlier ADR was directionally correct but too loose about checklist ownership and report scope, which leaves room for drift between the intended product boundary and placeholder owner-wide read models.

## Decision

1. `Project` remains the top-level workspace and ownership boundary.
   Every user-facing comparison and report experience belongs to one project workspace, not to an owner-wide portfolio by default.

2. `Property` remains a manual-entry child of a project.
   A property owns listing metadata only: address, listing price, property type, and source. It does not own checklist state or report state directly.

3. `Visit` is the only mutable field-capture aggregate in v1.
   A visit owns visit date, completion status, required checklist responses, red flags, recommendation notes, and attachment references. The write flow is `draft -> completed`, and completion is blocked until all required sections are present.

4. The checklist is application-owned configuration in v1, not a user-editable entity.
   The required sections are `property`, `building`, and `neighborhood`. `redflags` remains optional narrative capture. Responses are stored inside the visit aggregate as keyed score/note maps rather than in separate normalized checklist tables.

5. Comparison and report are read projections, not mutable source-of-truth records.
   Comparison reads the latest relevant visit per property within one project. Report reads a completed visit plus its property metadata. V1 does not introduce a first-class `reports` table or generated document store.

6. Browser-local autosave stays in the web client boundary only.
   Local storage keeps draft form state and attachment names for same-browser recovery. It is never the system of record, does not restore file binaries, and does not imply offline queueing or cross-device recovery.

## Service and module boundaries

### Web

- `apps/web/src/app/projects/**` and `apps/web/src/app/properties/**`
  Workspace navigation and manual project/property entry.
- `apps/web/src/app/visits/**` and `apps/web/src/components/visits/**`
  Checklist workspace, local autosave/restore, completion gating UI, and attachment retry UX.
- `apps/web/src/app/comparison/**`, `apps/web/src/app/report/**`, and `apps/web/src/components/report/**`
  Read-only presentation surfaces over report API payloads.
- `apps/web/src/components/auth/**`
  Session presentation only. Permission-model changes are outside this ADR's implementation scope.
- `apps/web/src/app/ops/**` and `apps/web/src/components/ops/**`
  Admin-only operational views; not part of the buyer workflow boundary.

### API

- `services/api/app/api/routes/projects.py`
  Owner-scoped CRUD for projects.
- `services/api/app/api/routes/properties.py`
  Owner-scoped CRUD for properties inside a project.
- `services/api/app/api/routes/visits.py`
  Visit creation, checklist persistence, completion invariants, and attachment ingestion. This route owns field-capture rules and must not own report formatting concerns.
- `services/api/app/api/routes/reports.py`
  Read-only projections for comparison and report payloads. This route must not mutate visit state.
- `services/api/app/core/auth.py` and `services/api/app/api/routes/auth.py`
  Cross-cutting session and resource access concerns. Any collaboration, public share-link, or role-model change requires a separate escalation.
- `services/api/app/services/storage.py`
  Binary storage adapter only. Metadata ownership stays in the API and database layers.

## Data model impact

| Entity | Responsibility | v1 impact |
| --- | --- | --- |
| `users` / `sessions` | Cross-cutting auth/session | Existing owner-scoped access model remains in place. |
| `projects` | Workspace and ownership root | Holds `owner_user_id`, `name`, `region`, `budget`, and `notes`. |
| `properties` | Candidate listing metadata | Holds `project_id`, `address`, `listing_price`, `property_type`, and `source`. |
| `visits` | Core field-capture aggregate | Holds `property_id`, `visit_date`, `status`, `red_flags`, `recommendation_notes`, `section_scores`, and `section_notes`. |
| `attachments` | Binary reference metadata | Holds `visit_id`, `filename`, `content_type`, `category`, `storage_backend`, `storage_key`, and `size_bytes`. |

Additional v1 constraints:

- No normalized `checklist_sections` table.
- No persisted `reports` table or PDF export artifact.
- Any move toward editable checklist templates, report snapshot storage, or analytics-oriented normalization is a new schema decision and requires a separate ADR.

## Key invariants

- Project ownership determines access to descendant properties, visits, attachments, comparison views, and reports.
- A property belongs to exactly one project.
- A visit belongs to exactly one property.
- The v1 required section keys stay stable. Renaming or removing them is a schema and analytics affecting change and must be escalated.
- A visit cannot be marked complete until every required section contains both a score and a note.
- Attachment upload failure must not erase already-saved visit data.
- Comparison and report outputs must be reproducible from persisted property and visit data.
- Owner-wide "latest report" or comparison placeholders are not the final product contract. Before staging sign-off, read models must align to project- or visit-scoped user workflows.

## Consequences

- The delivery path stays bounded around one write aggregate, `Visit`, and two derived read models, comparison and report.
- Checklist and report evolution after v1 will require explicit schema and contract decisions rather than incidental drift.
- Frontend and backend must keep section keys synchronized until a shared contract is introduced.
